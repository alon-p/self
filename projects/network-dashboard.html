Network Traffic Dashboard

One view for network health: pulling three disconnected data sources together, with an AI layer that analyzes your graphs and answers questions about them.

Stack
Python/Django · React · PostgreSQL · InfluxDB · Redis · Heroku · OpenAI · Auth0 · Microsoft Entra
Role
Backend developer & DevOps (2-developer team)
Context
Built for a startup, to their product spec

The Challenge

A startup wanted a single product for monitoring network health, but the data that mattered lived in three different systems that were never designed to be read together:

  • InfluxDB holds the raw monitoring time-series: the actual health metrics flowing in from the network.
  • NetConf holds the configuration behind that monitoring: which nodes exist and what each one is responsible for.
  • PostgreSQL holds the user side: what each user wanted to see, and which source each view should draw from.

On their own, each source answered only part of the question. Real insight meant joining a metric (Influx) to the node that produced it (NetConf) and to the view a user had set up (Postgres), across systems with completely different shapes and query models. The startup had the product vision; they needed someone to build it to their needs.

The Approach

I built the backend as the consolidation layer that sits in front of all three sources and presents one coherent view to the React front end. Rather than forcing the data into a single store, the backend queries each source for what it does best (time-series from Influx, topology/config from NetConf, user preferences and view definitions from Postgres) and stitches the results together per the user's configured view. Redis backs caching and the heavier coordination work so the dashboard stays responsive.

Key Technical Decisions

Keep each source authoritative, consolidate at the API

Instead of a fragile ETL into one database, the backend reads each system in its native model and composes the answer. This kept the monitoring data fresh and avoided a brittle sync pipeline.

A reproducible, one-click deploy story on Heroku

I set up a formal, push-button way to spin up new Heroku apps reproducibly, so a fresh environment could be stood up the same way every time, rather than hand-assembled. This made new deployments and environments predictable instead of one-off.

Flexible authentication per deployment

Different deployments needed different login options, so I set up Auth0 with multiple connections (Microsoft Entra federated, and an Auth0-managed user database), letting each deployment use the auth method that fit it.

The AI Layer

The standout feature: a user can take a graph (or several) of data from the system, hand it to the AI layer, and get it analyzed, then chat about it. Instead of staring at a chart and guessing, they can ask the assistant to interpret it, surface patterns, and help them find what they're looking for in the data. It turns a static visualization into something you can interrogate in plain language. Built on OpenAI, integrated into the same backend.

My Role

I was one of two developers. I owned the backend and the infrastructure end-to-end while a teammate built the React front end. Specifically, I built and managed:

  • The connections to all three data sources and the consolidation logic.
  • Database migrations.
  • CI/CD and the deployment workflow.
  • DevOps on Heroku, including the reproducible one-click app-creation process.
  • The OpenAI-backed analysis/chat integration.
  • Authentication via Auth0, with Microsoft Entra federation and an Auth0-managed database to support different deployments.

Results & Impact

The startup got the unified product they envisioned: one dashboard where network health, the configuration behind it, and each user's chosen view come together, plus an AI assistant that makes the data conversational rather than something you have to decode by hand. The reproducible Heroku deploy process meant new environments could be created reliably and repeatably instead of being assembled by hand each time.

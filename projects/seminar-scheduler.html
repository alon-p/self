Seminar Scheduling Automation

Generating two ready-made scheduling options from monday.com data and emailing them as a spreadsheet — alongside an error sheet that flags the data problems to fix.

Stack
Python · monday.com GraphQL API · Excel · email · Heroku
Role
Sole developer
Context
A small education NGO scheduling its educators

The Challenge

A small NGO that works with educators had a genuinely hard monthly job: scheduling who runs which seminar. It was a whole Sudoku — a tangle of the organization's own rules and constraints on one side, and the educators' availability and preferences on the other — and it was done by hand. Every cycle meant a lot of manual work: drafting, fixing, shifting things around, and reworking the whole board when one piece moved.

The real difficulty wasn't writing code — it was first understanding their rules and constraints, and how to balance them against what the educators needed.

The Approach

After working through the details with them, we decided not to chase a fully automatic answer, but to build a script that produces two complete scheduling options each run — enough to remove most of the manual effort while leaving the humans in charge of the final call. Scheduling people is never fully automatable: someone gets sick, something gets cancelled the morning of. The goal was to do the heavy lifting, not to pretend the messy last mile away.

The script pulls the data from monday.com via their GraphQL API, processes everything in memory, runs a scheduling formula with different parameters for each of the two options, and emails the result as a spreadsheet.

Key Technical Decisions

Two options instead of one fixed answer

The same formula run under different parameters yields two distinct schedules, giving the human schedulers room to choose and adjust rather than handing them a single take-it-or-leave-it result.

In-memory processing, no database

The job runs about once a month (occasionally twice), over a bounded amount of data — so standing up and maintaining a database would have been cost and complexity for no benefit. Joining everything in memory was the right-sized choice.

Email delivery, run from the Heroku interface — no web app

Rather than build and host an entire web application just to trigger a monthly script and show its output, I run it from the Heroku interface and deliver the result by email. It keeps hosting costs near zero for something that produces an output roughly once a month.

Handling Messy Real-World Data

The input lives in monday.com, maintained by people — so it's never perfectly clean. There are Hebrew typos, and items get moved around between runs. Instead of failing or silently producing a wrong schedule, the script emits an error worksheet alongside the result, pointing the team straight at the data problems to find and fix in monday. The tests cover these edge cases directly, which is how I trusted it to behave sensibly when the data didn't.

My Role

I owned it end to end: working out the scheduling rules with the client, integrating the monday.com GraphQL API, writing the in-memory scheduling logic and the two-option formula, generating the spreadsheet and error sheet, wiring up email delivery, and covering the edge cases with tests.

Results & Impact

  • Saved the team a lot of time and let them focus on the work that matters instead of wrestling a monthly scheduling puzzle.
  • Replaced a manual, error-prone scheduling process with two ready-made options generated in one run.
  • An error worksheet turns messy source data into a clear, fixable to-do list instead of a silent failure.
  • In use for two years with zero glitches — quiet, reliable automation that just does its job each cycle.

Reflection

Not every problem needs a database, a web app, or a UI. Matching the solution to the real shape of the problem — a bounded monthly job, humans still in the loop — meant a small, well-tested script could replace a recurring headache and keep running, untouched, for years.

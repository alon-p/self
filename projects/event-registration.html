Event Registration System — Legacy Modernization

Taking over a three-year-old, untested codebase running in production and bringing it up to modern standards — safer, cheaper to run, and finally releasable with confidence.

Stack
Node.js/Express · MySQL · AngularJS · AWS (VPC, ALB, EC2, CloudFront, CodePipeline) · GitHub Actions
Role
Developer leading the modernization
Context
Live event-registration service with real users

The Challenge

I inherited a roughly three-year-old codebase with all the hallmarks of code written under deadline pressure and then left behind:

  • Hardcoded SQL strings everywhere — queries assembled by hand throughout the code.
  • No real backend validation. The previous developer had used the front end as the gatekeeper; the backend methods trusted whatever reached them.
  • No tests at all.
  • One big monorepo containing both the Node.js backend and the AngularJS front end, tangled together.
  • An unmaintained Java/Tomcat service alongside it that nobody fully understood — we barely knew how it ran, let alone how to change it, and it carried a few known bugs.

All of this on a live service, so every change risked breaking something real users depended on.

The Approach

I modernized it in deliberate stages rather than attempting a risky big-bang rewrite: get visibility and safety first (tests, matching environments), then tackle the structural problems (the database, the Java service), then the release process.

Key Technical Decisions

Build a safety net before changing behavior

I introduced a test suite from scratch — going from 0 to roughly 200–300 tests — so refactoring the backend stopped being a guessing game. In the process I found and fixed several SQL injection vulnerabilities that the hardcoded-SQL-plus-no-validation combination had left exposed.

Make staging and production identical

They had drifted apart, which made debugging painful — issues showed up in one environment and not the other. Aligning the two removed a whole class of "works on staging" confusion.

Replace the Java/Tomcat service with Node modules

Since the Java service was unmaintained, barely understood, and buggy, the pragmatic move was to reimplement its behavior inside the Node app and retire it — consolidating onto a single runtime the team could actually maintain.

Migrate MySQL 5.7 → 8.0

Driven by cost and support: 5.7 was reaching end of extended support on AWS RDS, so upgrading both kept the database supported and reduced RDS costs. Done with minimal downtime on the live service.

Building the First CI/CD Pipeline

Before me, releases were entirely manual. I built the project's first real pipeline, and it evolved as the tooling did:

  • Started with Bitbucket Pipelines + CircleCI, then migrated to GitHub Actions, and from there to AWS (CodePipeline).
  • The front end now has a proper CI/CD pipeline that deploys to AWS automatically.
  • The Node backend has a test suite that GitHub Actions runs on every change; its deploy is still done manually today, but it's backed by a documented script that captures every step — so it's repeatable, not improvised.

My Role

I led the modernization end-to-end: the test suite, the security fixes, the database migration, retiring the Java service, building out CI/CD across both front and back end, and bringing the environments into parity.

Results & Impact

  • A previously untested, injection-prone codebase gained a 200–300 test safety net and had its SQL injection holes closed.
  • The app consolidated onto a single Node runtime, dropping the unmaintainable Java/Tomcat service.
  • The database moved to MySQL 8.0, staying supported and cutting RDS costs, with minimal downtime on a live system.
  • Releases went from fully manual to automated (front end) and documented + test-gated (backend).
  • Staging and production parity made bugs far easier to reproduce and fix.

What I'd Do Differently

I'd have invested earlier in containerizing the Node backend. That's the missing piece that would have let me automate its deploy fully instead of relying on the documented-manual process — and it's the natural next step for this project.

About

I'm a backend-leaning full-stack developer with 12+ years of experience, mostly in Python/Django and Node.js. What defines how I work is ownership — I want to understand how a system works, keep it under my control, and be the one who carries it all the way to production.

I'm drawn to the complicated parts other people would rather avoid: untangling mismatched data sources, modernizing legacy services, and building the back-office tools and automations that keep an organization running. I'm comfortable taking responsibility for the whole path — CI/CD, integrations, deployments, and production operations — not just the code that's fun to write.

I've done this across network monitoring, event management, and HR for a range of clients, and I bring the same instinct to each: make it work, make it understandable, and make sure I didn't break anything on the way.

Django Core

Six years of Django, and the framework I reach for on every project I can. I've also mentored several juniors on how to use it well, and I work with it mostly alongside Django REST Framework. Testing

Testing runs through almost everything I do — it's how I move through legacy code and modernize without breaking what works. I'd rather test the complicated scenarios in code than reproduce them by hand, and I wire the suite into the pipeline so it runs on every pull request and before every deploy. With good tests, refactoring stops being scary — and that's only more true with AI agents: solid tests let me see what a refactor actually changed and keep the result verified rather than one big pile of slop.

Django test framework Core

My testing home. I love that it stands up a real test database from your migrations and resets it between tests — so writing meaningful, DB-backed integration tests is effortless and still fast. I use clock-faking to cover time-dependent behaviour, too.

How I Work

01I take ownership

I like to own what I work on end-to-end. I'm not comfortable with parts of the system being a black box — I want to understand how everything works and keep it under control, from the database to the deploy.

02Understand before I change

When I get handed a messy codebase, the first thing I do is figure out what's actually going on. Before touching anything, I want a real picture of how it runs.

03Document the setup

As I learn a system, I write down how to set it up. Turning tribal knowledge into documentation makes the project maintainable — for me and for whoever comes next.

04Add tests so I know I didn't break anything

I add tests before and as I modernize, because I genuinely need to know I haven't broken something that was working. Tests are how I move through legacy code with confidence instead of crossing my fingers.

05Modernize pragmatically

Once I understand it and it's covered, I modernize whatever I can. I favor the tool that solves the problem well over the one that looks impressive — the right call for the client beats the shiny one.
